## 2. 现状诊断

### 2.1 当前点击区域实现的问题

当前项目中的 Live2D 点击拖动逻辑并不是按皮套真实可见轮廓判断，而是人为覆盖了一个矩形交互区域：

1. `live-2d/js/model/model-interaction.js` 中，`updateInteractionArea()` 将交互区写死为：
   - `interactionWidth = model.width / 3`
   - `interactionHeight = model.height * 0.7`
2. `containsPoint()` 被重写为矩形命中判断，而不是模型真实可见像素命中。
3. `live-2d/js/model/model-setup.js` 中的 `setupHitTest()` 也继续沿用这个矩形交互区。
4. `live-2d/js/ui/ui-controller.js` 的鼠标穿透控制又依赖 `currentModel.containsPoint()`，因此整个窗口穿透逻辑同样建立在粗糙矩形上。

这会直接导致两个问题：

1. 点击皮套外的空气区域也可能被判定为命中。
2. 当皮套部分移出屏幕时，仍然可见的头发、呆毛、头部边缘等位置可能无法重新抓取。

### 2.2 当前位置与缩放体系的问题

当前项目的默认位置和保存位置策略也存在明显不稳定性：

1. `setupInitialModelProperties()` 默认直接使用 `x = window.innerWidth * 1.35`、`y = window.innerHeight * 0.8`。
2. 保存位置时使用 `model.x / window.innerWidth` 与 `model.y / window.innerHeight`，但默认值本身可能大于 1，说明这套值并不是真正稳定的标准化坐标。
3. 当前保存的是模型左上角近似位置，而不是稳定的视觉锚点位置。
4. 屏幕尺寸变化时，仅靠当前 `x/window.innerWidth` 的方式无法保证不同分辨率和多屏布局下的位置感受一致。

### 2.3 当前多屏移动实现的问题

当前多屏限制的根源在主进程窗口架构，而不只是拖动逻辑：

1. `live-2d/main.js` 创建窗口时只使用 `screen.getPrimaryDisplay()` 的 `workAreaSize`。
2. BrowserWindow 初始大小仅覆盖主屏。
3. `will-move` 中又把窗口边界限制在主屏工作区内。
4. `window-move` IPC 也只按主屏宽高做限制。

虽然当前 Live2D 模式实际主要是在窗口内部拖动模型，而不是拖动整个窗口，但由于透明窗口只覆盖主屏，因此模型无论如何也不可能真正移动到另一个屏幕。

## 3. 参考项目结论

### 3.1 来自 `reference-projects/2D/testdrag.py` 的借鉴

该参考实现最有价值的点在于：

1. 它不依赖粗矩形判断，而是通过 `glReadPixels()` 读取点击位置的 Alpha 通道。
2. 只要像素不透明，就判定鼠标命中了模型真实可见区域。
3. 这种方案天然适配不同模型轮廓，不需要为每个模型重新人工画矩形。
4. 它能很好解决“只剩半个头或呆毛露在屏幕内时无法抓回”的问题。

结论：**精确点击区域的核心方向应采用“按最终渲染透明度进行像素级命中判定”的思想。**

### 3.2 来自 `reference-projects/N.E.K.O/static/common-ui-hud.js` 的借鉴

该参考实现最有价值的点在于：

1. 它通过 `getAllDisplays()` 获取所有显示器边界。
2. 它根据拖动对象中心点所在位置，判断当前对象属于哪块屏幕。
3. 它用当前屏幕边界做约束，而不是写死主屏。
4. 它会在窗口大小变化后重新校准位置，并保存本地位置偏好。

结论：**多屏支持应采用“显示器集合 + 当前中心点归属屏幕 + 边界重算”的设计方式。**

### 3.3 来自 `reference-projects/N.E.K.O/config/__init__.py` 的借鉴

其配置注释明确指出：`position / scale / viewport / display` 应作为本地偏好存储，而不是绑定到角色定义本身。

结论：**位置、缩放、显示器归属等信息应作为本地布局状态单独管理，不应和模型资源定义硬耦合。**

## 4. 设计目标

本方案设计要达到以下目标：

1. 皮套点击区域尽量贴合最终可见轮廓，而不是矩形近似。
2. 切换不同 Live2D 模型时，点击判定仍能自动适配。
3. 皮套可以在双屏或多屏环境下自由跨屏移动。
4. 位置和缩放在不同分辨率、不同主副屏排列下保持更稳定的视觉结果。
5. 保持现有架构可演进，优先复用当前 `model-interaction.js`、`ui-controller.js` 和 `main.js`，避免一次性整体推翻。

## 5. 总体方案

本次实现方案拆为三层：

1. **精确命中层**：用渲染结果 Alpha 命中替代当前硬编码矩形。
2. **布局状态层**：引入与屏幕相关的统一坐标、缩放和锚点保存方式。
3. **多屏窗口层**：把透明窗口从“只覆盖主屏”升级为“覆盖虚拟桌面”，从而允许皮套跨屏移动。

优先级顺序：

1. 先完成精确命中层。
2. 再完成布局状态层。
3. 最后接入多屏窗口层并完成联调。

## 6. 需求一实现方案：点击区域精确化

### 6.1 方案选型对比

#### 方案 A：继续使用矩形或手工调矩形

不采用。原因：

1. 无法覆盖复杂发型、姿态和轮廓。
2. 每切换模型都可能要重新调参。
3. 无法真正解决“点空气误拖”和“只剩局部可见却抓不到”的问题。

#### 方案 B：依赖 Live2D 自带 HitArea

不作为主方案，只作为辅助信息。原因：

1. 不同模型的 HitArea 质量不稳定。
2. 很多模型的 HitArea 偏语义区域，如头、身体，不等于完整可拖动轮廓。
3. 无法保证“完美覆盖”目标。

#### 方案 C：基于最终渲染 Alpha 的像素级命中

采用，作为主方案。原因：

1. 与模型实际显示结果一致。
2. 天然适配不同皮套轮廓。
3. 只要屏幕里还有可见像素，就仍能命中。
4. 参考 `2D/testdrag.py` 已证明该方向有效。

### 6.2 设计细节

新增一个独立的命中检测模块，建议命名为：

- `live-2d/js/model/model-hit-detector.js`

职责：

1. 将鼠标 client 坐标转换为 PIXI 渲染坐标。
2. 读取该位置的渲染 Alpha。
3. 用阈值判断该位置是否命中模型可见像素。
4. 在需要时做小范围邻域采样，提升细发丝、半透明边缘的可点击性。

### 6.3 命中算法

建议算法如下：

1. 先做粗判断：若鼠标不在模型可见包围盒附近，则直接返回未命中，避免频繁读像素。
2. 若通过粗判断，再执行精确判断：
   - 将 `clientX/clientY` 转换到 canvas 内部像素坐标。
   - 从渲染结果读取该像素 Alpha。
   - 若 Alpha 大于阈值，则命中。
3. 为避免发丝边缘过于苛刻，可加 3x3 或 5x5 小邻域采样：
   - 只要邻域内存在 Alpha 超阈值像素，即认为命中。
4. 拖动开始后进入“拖动态锁定”：
   - 鼠标按下命中后，直到 `mouseup` 之前都保持拖动，不再重新判定像素。

### 6.4 性能策略

像素读取属于高成本操作，因此需要控制频率：

1. 只在 `mousemove`、`mousedown` 的必要时机触发。
2. 鼠标静止时不重复采样。
3. 每帧最多执行一次精确采样，可用 `requestAnimationFrame` 节流。
4. 命中检测优先走“粗包围盒 -> 精确 Alpha”双层判断。

### 6.5 对现有代码的改造点

#### `live-2d/js/model/model-interaction.js`

需要做以下调整：

1. 删除当前 `updateInteractionArea()` 的硬编码矩形主逻辑。
2. `containsPoint()` 改为：
   - 先判断聊天框等 UI。
   - 再调用新的精确命中模块判断模型区域。
3. `mousedown`、`mousemove`、`click`、`wheel` 的判定统一改为使用精确命中结果。
4. 保留拖动态锁定，避免拖动中断。

#### `live-2d/js/model/model-setup.js`

需要做以下调整：

1. 移除当前 `setupHitTest()` 中基于矩形的默认逻辑。
2. 当前实现中不再额外注入旧 `hitTest` 矩形代理逻辑，避免再次引入与精确命中不一致的判断链路。

#### `live-2d/js/ui/ui-controller.js`

需要做以下调整：

1. 鼠标穿透控制改为调用 `global.modelController.isInteractiveClientPoint()`。
2. UI 层统一通过交互控制器判断“模型可点击区域 + 聊天框区域”，不再直接依赖旧矩形命中逻辑。

### 6.6 预期效果

完成后可达到：

1. 点到皮套真实显示像素才可拖动。
2. 点到皮套周围空气区域不会误拖。
3. 皮套只剩半个头或呆毛可见时，也可以通过这些区域抓回。
4. 切换不同皮套后无需重新调矩形热区。

## 7. 需求二实现方案：多屏自由移动

### 7.1 核心架构决策

当前 Live2D 是在一个透明 Electron 窗口中渲染模型，并在窗口内部拖动模型位置。因此要支持跨屏，不应再让这个透明窗口只覆盖主屏。

采用方案：

**将 BrowserWindow 的覆盖范围扩展到整个虚拟桌面。**

也就是：

1. 主进程不再只取主屏尺寸。
2. 改为读取所有显示器边界，并计算一个总的虚拟桌面包围矩形。
3. 透明窗口的大小和位置覆盖整个虚拟桌面。
4. 模型位置在这个虚拟桌面坐标系里移动，因此自然可以跨屏。

### 7.2 主进程设计

#### `live-2d/main.js`

需要新增一个显示器布局计算函数，例如：

- `getVirtualDesktopBounds()`

职责：

1. 调用 `screen.getAllDisplays()`。
2. 计算所有 `display.bounds` 的并集。
3. 返回 `{ x, y, width, height }`。

窗口创建改为：

1. `BrowserWindow.width/height` 使用虚拟桌面宽高。
2. `win.setPosition(virtualBounds.x, virtualBounds.y)`。
3. 去掉当前基于主屏的 `will-move` 限制逻辑，或者改成基于虚拟桌面的限制。
4. 监听 `display-added`、`display-removed`、`display-metrics-changed`，在显示器拓扑变化后重新刷新窗口范围。

### 7.3 渲染层设计

由于窗口原点不再固定在主屏左上角，渲染层需要认知“虚拟桌面坐标”和“窗口内局部坐标”的关系：

1. 浏览器窗口左上角对应虚拟桌面 `virtualBounds.x / virtualBounds.y`。
2. 渲染层内的鼠标坐标仍是窗口局部坐标。
3. 若后续需要知道模型属于哪块屏幕，需要把局部坐标转换为虚拟桌面绝对坐标。

### 7.4 屏幕归属判定

借鉴 N.E.K.O 的思路，新增“当前模型归属显示器”判定：

1. 取模型视觉锚点中心点。
2. 将其转换为虚拟桌面绝对坐标。
3. 在 `screen.getAllDisplays()` 中找到包含该点的显示器。
4. 该显示器作为当前显示器，用于默认位置、边界约束和后续保存。

### 7.5 边界策略

当前实现中的多屏策略以“虚拟桌面全覆盖 + 中心点归属显示器” 为主，不再把模型限制在主屏：

1. 允许模型在虚拟桌面范围内自由跨过屏幕边界。
2. 保存布局时按模型中心点所在显示器写入 `display_id` 与相对坐标。
3. 当前版本不额外附加“最小可见边距”约束，以避免拖动过程中出现新的强制吸附或跳动。

这样可以同时满足：

1. 模型自由跨屏。
2. 多屏保存与恢复使用统一的显示器归属坐标系。

## 8. 坐标与缩放体系设计

这是本次设计中的关键增强点，用于解决“不同屏幕大小下默认值不稳定”的问题。

### 8.1 设计原则

1. 不再把模型左上角直接当成稳定位置基准。
2. 引入“视觉锚点”概念，统一保存和恢复。
3. 默认位置应参考屏幕尺寸与工作区中心，默认缩放先保持显式保存/恢复策略，而不是继续写死旧常量坐标。
4. 位置和缩放信息作为本地布局偏好存储。

### 8.2 视觉锚点选择

本次实现采用：

**中心锚点**

原因：

1. 当前代码中的多屏归属判断和默认定位都已基于模型中心点实现。
2. 固定中心坐标策略与中心锚点天然一致，默认恢复逻辑更直接。
3. 相比继续沿用左上角坐标，中心锚点在跨屏保存和恢复时更稳定。

### 8.3 位置保存格式

当前实现保存的布局对象如下：

```json
"ui": {
  "model_layout": {
    "version": 1,
    "anchor_type": "center",
    "reference_space": "display-workarea",
    "display_id": 123456,
    "anchor_x_ratio": 0.5,
    "anchor_y_ratio": 0.5,
    "saved_display_width": 2560,
    "saved_display_height": 1440,
    "scale": 2.3
  }
}
```

字段说明：

1. `display_id`：上次保存时锚点所在显示器。
2. `anchor_x_ratio` / `anchor_y_ratio`：锚点在该显示器工作区内的相对位置。
3. `saved_display_width` / `saved_display_height`：用于屏幕变化时恢复比例。
4. `scale`：保存时的基础缩放值。

### 8.4 默认位置策略

默认位置不再使用 `1.35` 这种硬编码倍数，改为固定中心坐标策略：

1. 默认归属主显示器。
2. 默认中心点位于主显示器工作区中心。
3. 模型实际位置通过 `centerX - model.width / 2` 与 `centerY - model.height / 2` 计算得到。

本项目当前采用：

**主显示器工作区固定中心坐标**

原因：

1. 与当前 `ModelLayoutManager.applyDefaultCenterLayout()` 的实现完全一致。
2. 默认落点稳定，不再出现旧版本 `x > 1` 的异常比例坐标。
3. 在单屏和多屏下都能保证首次加载位置可预期。

### 8.5 当前缩放策略

当前实现中的缩放恢复策略为：

1. 若 `model_layout.scale` 存在且有效，则直接恢复该缩放值。
2. 若不存在保存布局，则使用 `ui.model_scale` 或默认值 `2.3` 作为初始缩放。
3. 本次实现暂不引入按显示器高度自动换算缩放的逻辑。

这样做的原因是：

1. 先保证位置系统和多屏保存恢复链路稳定。
2. 避免在同一轮改动里同时引入位置和缩放双重自适应，增加联调复杂度。
3. 为后续单独评估“按屏幕比例缩放”留下清晰扩展点。

### 8.6 向后兼容策略

当前已有旧字段：

- `ui.model_position.x`
- `ui.model_position.y`
- `ui.model_scale`

当前实现采用“新旧字段并存保存、默认中心回退”的兼容策略：

1. 若存在有效的 `ui.model_layout`，优先读取新布局数据。
2. 若不存在有效布局，则直接回退到主显示器工作区固定中心坐标。
3. 每次保存新布局时，仍同步回写 `ui.model_position` 与 `ui.model_scale`，保留旧字段。
4. 当前版本不执行旧字段到新布局的自动迁移，避免把历史异常坐标再次写入新结构。

## 9. 模块拆分建议

### 9.1 新增模块

#### `live-2d/js/model/model-hit-detector.js`

职责：

1. 模型真实可见像素命中判定。
2. 包围盒粗判与 Alpha 精判。
3. 可选邻域采样。

#### `live-2d/js/model/model-layout-manager.js`

职责：

1. 统一处理位置、锚点、缩放恢复与保存。
2. 屏幕变化时重新计算布局。
3. 基于模型中心点判断显示器归属，并生成布局快照。

#### `live-2d/js/core/display-layout.js` 或主进程辅助函数

职责：

1. 获取所有显示器信息。
2. 计算虚拟桌面范围。
3. 根据给定点确定当前显示器。

### 9.2 现有模块责任调整

#### `model-interaction.js`

聚焦：

1. 交互流程。
2. 拖动状态。
3. 与命中模块、布局模块协作。

#### `model-setup.js`

聚焦：

1. 模型加载。
2. PIXI 初始化。
3. 命中模块和布局模块接入。

#### `main.js`

聚焦：

1. 多显示器窗口范围管理。
2. 屏幕信息 IPC 暴露。
3. 配置存储。

## 10. 实施顺序建议

### 阶段 1：点击区域重构

1. 新增 `model-hit-detector.js`。
2. 用精确命中替换 `model-interaction.js` 当前矩形逻辑。
3. 联动 `ui-controller.js` 的鼠标穿透。

## 11. 验收方案

### 11.1 点击区域验收

1. 模型周围空气点击不触发拖动。
2. 点击头发、呆毛、身体边缘可正常拖动。
3. 模型部分移出屏幕后仍可从剩余可见区域拖回。
4. 切换至少 2 个不同轮廓模型后，命中效果仍然稳定。

### 11.2 多屏验收

1. 双屏环境下模型可从屏幕 A 拖到屏幕 B。
2. 跨屏后仍可继续点击并拖动。
3. 模型不会被主屏边界强行拦截。
4. 显示器分辨率不同情况下，模型位置恢复基本符合预期。

### 11.3 布局验收

1. 更换分辨率后默认位置不再严重偏离。
2. 存在 `model_layout` 时，可按保存的显示器和中心点比例恢复位置。
3. 缺少 `model_layout` 时，系统回退到固定中心坐标，不导致模型直接消失。

## 12. 风险与应对

### 12.1 Alpha 读取性能风险

风险：频繁读像素可能影响帧率。

应对：

1. 加粗判。
2. 节流到每帧一次。
3. 仅在交互相关事件中采样。

### 12.2 多屏透明窗口兼容风险

风险：Electron 透明窗口覆盖整个虚拟桌面后，在部分环境可能出现焦点或性能问题。

应对：

1. 当前实现先保持主链路简洁，不额外叠加运行时 feature flag。
2. 若 `type: 'desktop'` 与多屏组合不稳定，可退回普通无边框透明窗口类型做兼容试验。
3. 如需灰度方案，后续再单独增加配置开关，而不是在本次恢复中继续扩大改动面。

### 12.3 旧配置迁移风险

风险：历史配置里只有旧 `model_position`、没有 `model_layout` 时，用户原先手动摆放的位置不会自动恢复。

应对：

1. 当前版本统一回退到主显示器工作区固定中心坐标，保证先可见、可拖动。
2. 新布局一旦保存后，即可进入新的多屏保存恢复链路。
3. 旧字段继续保留，便于后续如有需要再补充迁移脚本。

## 13. 回退策略

当前实现未引入运行时功能开关，回退策略采用代码级回退：

1. 回退 `live-2d/js/model/model-hit-detector.js` 与 `live-2d/js/model/model-interaction.js`，可恢复旧矩形命中逻辑。
2. 回退 `live-2d/js/model/model-layout-manager.js`、`live-2d/js/model/model-setup.js` 与 `live-2d/main.js`，可恢复旧单屏布局策略。
3. 若仅需清空新布局数据，可删除 `config.json` 中的 `ui.model_layout`，系统会在下次启动时回退到固定中心坐标。

## 14. 结论

本方案的核心设计结论如下：

1. **点击区域问题的正确解法不是继续调矩形，而是改为基于最终渲染 Alpha 的像素级命中。**
2. **多屏移动问题的正确解法不是只改拖动边界，而是把透明窗口扩展到虚拟桌面，并引入显示器归属判定。**
3. **位置问题不能继续依赖当前硬编码默认值，应改为基于中心锚点与固定中心坐标的布局状态管理；缩放则先采用直接保存和恢复策略。**
4. **参考项目中，`2D` 提供了精确命中的方向，`N.E.K.O` 提供了多屏边界与本地偏好管理的方向，两者结合后正好可以覆盖本项目两个需求。**

本方案现已与当前实现对齐，可作为本轮提交和后续验证的设计基线。
