# 文档书写规范（项目级）

- 适用范围：`my-neuro` 全仓库
- 版本：`v1.0.0`
- 状态：`生效`
- 生效日期：`2026-03-16`
- 维护人：`项目维护组`

## 1. 目标

本规范用于统一项目文档结构与书写方式，避免开发进度与文档脱节，确保：

1. 文档可追溯（需求、任务、代码、发布可关联）
2. 文档可维护（责任人明确、状态清晰、更新可审计）
3. 文档可用于版本管理（变更记录标准化）

## 2. 文档分类与最小产物

每项开发任务至少应具备以下文档之一：

1. 功能说明文档（Feature）
2. 技术设计文档（Design）
3. 变更记录文档（ChangeLog）
4. 发布说明文档（Release Note）

建议在 `document/` 下按类型建立子目录：

- `document/feature/`
- `document/design/`
- `document/changelog/`
- `document/release/`

## 3. 命名规范

文档文件名统一使用：

`<类型>-<主题>-<YYYYMMDD>-v<主>.<次>.<修>.md`

示例：

`feature-live2d-emotion-20260316-v1.0.0.md`

要求：

1. 文件名仅使用小写英文、数字、连字符 `-`
2. 禁止空格与中文文件名
3. 主题需可读且可检索，避免 `test`、`new`、`final` 等无语义命名

## 4. 文档头部元数据（必填）

每个文档开头必须包含以下字段：

```md
# 文档标题

- 文档ID：DOC-<类型>-<唯一编号>
- 类型：Feature | Design | ChangeLog | Release
- 状态：Draft | Reviewing | Approved | Implemented | Deprecated
- 当前版本：vX.Y.Z
- 创建日期：YYYY-MM-DD
- 更新日期：YYYY-MM-DD
- 作者：姓名/团队
- 评审人：姓名/团队
- 关联需求：REQ-xxx（可多项）
- 关联任务：TASK-xxx / Issue #xxx
- 关联代码：PR #xxx / Commit <hash>
- 关联发布：Release vX.Y.Z
```

说明：

1. `状态` 必须与实际阶段一致
2. `关联代码` 至少包含一个可定位链接（PR 或 Commit）
3. `更新日期` 每次内容变更后必须更新

## 5. 状态流转规则

统一状态机：

1. `Draft`：草稿，作者可编辑
2. `Reviewing`：评审中，不得进行破坏性改写
3. `Approved`：评审通过，可作为实施依据
4. `Implemented`：已落地并随代码发布
5. `Deprecated`：废弃，仅保留追溯

流转约束：

1. `Draft -> Reviewing -> Approved -> Implemented`
2. 任何状态可进入 `Deprecated`，但必须注明原因与替代文档

## 6. 版本管理规则

采用语义化版本：`v主版本.次版本.修订版本`

1. 主版本（Major）：结构性重写或语义重大变更
2. 次版本（Minor）：新增章节、补充关键信息
3. 修订版本（Patch）：错字修复、表述优化、非语义改动

每次更新必须追加“变更记录”章节：

```md
## 变更记录

| 日期 | 版本 | 修改人 | 变更类型 | 说明 |
|---|---|---|---|---|
| 2026-03-16 | v1.0.0 | xxx | 新建 | 初版建立 |
```

## 7. 可追溯性要求（强制）

每篇文档必须具备以下追溯链中的至少三段：

`需求 -> 任务 -> 代码 -> 测试 -> 发布`

示例：

1. 需求：`REQ-012`
2. 任务：`Issue #88`
3. 代码：`PR #143`
4. 测试：`tests/live2d/test_emotion.py`
5. 发布：`Release v2.4.0`

若暂缺某段，需显式标注：`待补充`，并在下一次版本更新中补齐。

## 8. 文档内容结构建议

推荐正文结构：

1. 背景与目标
2. 范围（包含/不包含）
3. 方案说明
4. 风险与回滚
5. 验收标准
6. 关联项（需求/任务/代码/测试/发布）

## 9. 评审与归档

1. `Approved` 之前不得作为最终实施依据
2. 每次发布前，需核对关键文档状态是否为 `Implemented` 或 `Approved`
3. 废弃文档移至 `document/archive/`（可选）并保留元数据与变更记录

## 10. 执行建议（落地）

1. 提交代码时，同步更新对应文档的“关联代码”和“变更记录”
2. 合并 PR 前，在描述中附文档路径
3. 版本发布时，将变更汇总至 Release Note
