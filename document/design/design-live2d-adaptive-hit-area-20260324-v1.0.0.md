## 1. 背景与目标

当前 `live-2d` 皮套模型的点击与拖拽命中区域，使用的是人工定义的矩形范围，而不是模型真实可见身体区域。

现状问题主要有两类：

1. 模型部分移出屏幕后，用户看到的头发、呆毛等仍在画面内，但由于不在矩形热区内，无法抓住可见部分把模型拖回
2. 模型周围存在“虚空可点区”，即鼠标没有点到皮套身体本身，也能在空气位置触发拖拽

这会直接影响：

1. 拖拽体验不自然
2. 穿透判断不准确
3. 切换不同体型、不同留白的 Live2D 皮套后，热区需要手工重调

本方案目标是：

1. 点击拖拽热区尽量贴合模型当前真实可见区域
2. 新旧皮套切换后自动适配，不依赖每个模型单独手调矩形参数
3. 与当前 Electron + PIXI + Live2D 单窗口架构兼容
4. 保持拖拽、点击穿透、滚轮缩放等交互能力可继续工作
5. 为后续多屏拖拽、窗口穿透、命中调试提供统一命中服务层

## 2. 范围

### 2.1 包含范围

1. Live2D 模型点击命中判定逻辑改造
2. 拖拽起始判定逻辑改造
3. 鼠标悬停时窗口穿透开关判定逻辑改造
4. 模型切换、缩放、动作变化时的命中区域自适应策略
5. 命中调试与回退策略设计

### 2.2 不包含范围

1. VRM 模式的命中逻辑重构
2. 聊天框拖拽规则重做
3. 多模型同时显示时的命中竞争
4. 非 Windows 平台专项优化
5. 本文档不直接包含最终实现代码

## 3. 现状问题分析

当前问题集中在 [live-2d/js/model/model-interaction.js](/d:/Github/my-neuro/live-2d/js/model/model-interaction.js)。

现状逻辑特点：

1. `updateInteractionArea()` 直接把命中区写死为 `model.width / 3` 与 `model.height * 0.7`
2. `containsPoint()` 被覆盖后，核心判定变成“点是否落在这个人工矩形里”
3. 该矩形与模型真实透明区域、发丝、饰品、呆毛、姿态变化没有直接对应关系
4. 模型缩放和动作会改变画面视觉外形，但不会改变热区语义

因此会出现：

1. 可见身体点不到
2. 不可见空气点得到
3. 不同模型的上下左右留白差异越大，问题越严重

## 4. 参考方案可借鉴点

你提供的 PyQt/OpenGL 示例，核心思路不是“按预设矩形命中”，而是：

1. 先把模型真正渲染出来
2. 再读取鼠标所在像素的 Alpha
3. 当该像素透明度高于阈值时，视为点中了模型

这个思路的价值在于：

1. 命中区域天然跟随模型真实可见轮廓
2. 不依赖某个皮套的固定宽高比例
3. 发丝、头顶装饰、身体边缘等都能自然参与点击判定

但在本项目中不能直接照搬，原因是：

1. 当前项目基于 Electron + PIXI，而不是 PyQt + 原生 OpenGL 控件
2. 如果在高频 `mousemove` 上直接逐点 `glReadPixels`，容易带来明显性能压力和渲染同步开销
3. 当前项目还有聊天框命中、窗口穿透、拖拽、多屏迁移等链路，需要统一纳入同一套判定服务

因此，本项目建议借鉴“按可见像素 Alpha 命中”的原则，但采用更适合当前栈的实现形式。

## 5. 新方案总览

本方案采用：

`基于模型可见像素 Alpha 的自适应命中 + 可选轮廓膨胀容错 + 统一命中服务层`

核心原则：

1. 模型是否可点，不再由固定矩形决定
2. 命中判定以“当前鼠标点对应的模型渲染结果是否为有效非透明像素”为主
3. 命中结果允许保留少量容错半径，避免发丝边缘过于苛刻
4. 命中逻辑统一服务于拖拽开始、鼠标穿透切换、点击动作触发
5. 当像素级判定不可用或异常时，允许退回到较保守的包围盒兜底

## 6. 方案说明

### 6.1 架构调整

建议新增命中服务模块：

1. `live-2d/js/model/model-hit-test-service.js`

职责：

1. 提供模型命中判定 `hitTest(point)`
2. 管理 Alpha 采样、阈值、容错半径与缓存
3. 对外屏蔽 PIXI 坐标、渲染纹理、缩放换算细节
4. 提供调试信息，例如最近一次命中点、Alpha 值、是否走兜底路径

职责归属建议：

`model-interaction.js`

1. 继续负责拖拽、滚轮、点击、跨屏联动
2. 不再自行维护矩形热区
3. 统一调用 `modelHitTestService.hitTest()`

`model-hit-test-service.js`

1. 负责像素级命中判定
2. 负责命中缓存与性能优化
3. 负责模型切换或尺寸变化后的命中刷新

### 6.2 命中判定策略

建议采用三级判定链路：

1. 第一层：聊天框与其他 UI 命中优先
2. 第二层：模型可见像素 Alpha 命中
3. 第三层：异常兜底命中

具体规则：

1. 若点在聊天框区域内，直接视为可交互，避免影响现有聊天框体验
2. 若点不在聊天框内，则进入模型 Alpha 命中判定
3. 若 Alpha 判定失败但出现渲染读回异常，可退回到较小的模型包围盒兜底
4. 默认不再使用当前的 `width/3 * height*0.7` 矩形逻辑

### 6.3 Alpha 命中实现建议

在 Electron + PIXI 场景下，建议优先采用“离屏渲染结果采样”而不是直接依赖当前矩形。

建议流程：

1. 以模型当前渲染结果为基础，生成可读取的命中纹理或使用现有渲染目标读取单点像素
2. 将鼠标点从窗口坐标转换到渲染坐标
3. 读取该点像素 Alpha
4. 当 Alpha 大于阈值时，认定为命中模型

阈值建议：

1. 默认阈值 `alpha >= 32`
2. 若发现边缘过于难点，可调到 `16`
3. 若发现半透明发丝区域误触过多，可调到 `48`

这样可以兼顾：

1. 点击头发、呆毛、饰品等细小可见区域
2. 避免在透明背景中凭空拖动模型

### 6.4 边缘容错策略

像素级命中如果过于严格，可能造成：

1. 发丝边缘难以点击
2. 半透明抗锯齿区域不稳定

因此建议增加轻量容错：

1. 以鼠标点为中心，检测一个很小的采样邻域，例如 `3x3` 或 `5x5`
2. 只要邻域内存在一个 Alpha 超过阈值的像素，即视为命中
3. 该容错只用于点击与拖拽起始，不用于大范围吸附

这样可以保留“贴合身体”的核心体验，同时降低边缘点击挫败感。

### 6.5 缓存与刷新策略

为了避免每次鼠标移动都做高成本全量计算，建议采用按需刷新。

建议刷新时机：

1. 模型首次加载完成后
2. 切换皮套后
3. 模型缩放后
4. 窗口尺寸变化后
5. 模型位置显著变化后
6. 触发关键动作导致外形明显变化时，可按需刷新

建议策略：

1. 鼠标移动时只做单点或小邻域采样
2. 不要在每帧重建完整命中图
3. 若实现上需要缓存命中纹理，则在上述时机增量刷新

### 6.6 与拖拽链路的衔接

拖拽规则建议修改为：

1. `mousedown` 只有在 `hitTest()` 返回命中时才进入拖拽状态
2. 拖拽开始后，后续 `mousemove` 不再要求鼠标持续命中模型实体，避免拖拽过程中因鼠标略离开身体而中断
3. `mouseup` 后重新回到实时命中判定

这可以解决：

1. 起拖必须点到真实身体
2. 拖动过程中不因为细小边缘抖动而失去控制

### 6.7 与窗口穿透的衔接

当前窗口穿透逻辑本质上依赖“鼠标是否在模型可点区域内”。

因此建议同步改造：

1. `mouseover` / `mouseout` 不再依赖矩形命中
2. 全局鼠标移动时，统一调用 `hitTest()` 来决定 `set-ignore-mouse-events`
3. 若鼠标位于模型透明区，则窗口保持穿透
4. 若鼠标位于模型可见身体区或聊天框区域，则关闭穿透

这样可以同步解决“空气区域也拦鼠标”的问题。

### 6.8 与换皮套自动适配的关系

本需求最核心的非功能目标之一，是“切换新皮套时无需重新手调命中区”。

Alpha 命中方案天然具备这一优势，因为：

1. 它依赖的是当前渲染结果，而不是某个模型预设尺寸比例
2. 不同模型即便留白不同、头发更高、身体更窄，也会自动按真实显示区域参与命中
3. 只要模型能正确渲染，命中区域就能自然适配

因此不建议继续为每个模型增加独立命中配置文件，除非后续确认个别模型确实需要特殊覆盖。

### 6.9 调试能力建议

建议加入一个调试开关，例如：

1. `ui.hit_test_debug_enabled`

开启后可提供：

1. 在日志中打印点击点坐标、Alpha 值、命中结果、是否走兜底
2. 可选显示一个小型命中点标记
3. 可选输出当前模型的包围盒、采样阈值与缩放比例

这样在更换皮套或处理异常模型时更容易定位问题。

### 6.10 回滚策略

建议保留配置开关：

1. `ui.adaptive_hit_test_enabled`

规则：

1. 开启时使用新方案
2. 关闭时退回旧矩形命中逻辑
3. 新方案联调期间默认建议保留双轨支持，避免一次性替换后难以排障

## 7. 实施步骤建议

### 7.1 第一阶段：命中服务抽离

1. 新增 `model-hit-test-service.js`
2. 把当前矩形命中逻辑从 [live-2d/js/model/model-interaction.js](/d:/Github/my-neuro/live-2d/js/model/model-interaction.js) 中拆出
3. 保留旧逻辑作为 fallback

### 7.2 第二阶段：接入拖拽与穿透

1. `mousedown`、`click`、`mouseover`、窗口穿透判断统一改为调用命中服务
2. 确保聊天框区域仍优先可交互
3. 确保拖拽开始后不会被实时命中抖动打断

### 7.3 第三阶段：性能与容错调优

1. 调整 Alpha 阈值
2. 增加小邻域采样容错
3. 根据模型缩放和高 DPI 情况校准坐标转换

### 7.4 第四阶段：模型切换与回归

1. 验证不同体型、不同留白的皮套切换后命中是否自动适配
2. 验证模型部分移出屏幕后，用户仍能抓住可见头发或身体把它拖回
3. 验证模型周边空气区域不再误触

## 8. 风险与回滚

### 8.1 主要风险

1. 直接读取像素 Alpha 可能带来性能开销，尤其是高频鼠标移动场景
2. 高 DPI 或缩放换算不一致时，可能出现视觉点中但采样点偏移
3. 半透明区域过多的模型，阈值过低时可能扩大误触范围
4. 读回渲染结果时若与当前渲染管线冲突，可能出现偶发命中抖动

### 8.2 风险缓解

1. 只做单点或小邻域采样，不做高频全图扫描
2. 统一封装窗口坐标到渲染坐标的转换方法
3. 提供阈值配置与调试开关
4. 保留旧矩形逻辑作为 fallback
5. 优先在常用皮套上做人测，再扩大验证范围

### 8.3 回滚策略

1. 保留 `ui.adaptive_hit_test_enabled`
2. 若新方案出现性能或命中异常，立即退回旧矩形命中逻辑
3. 调试期不要删除旧逻辑，只降低其优先级

## 9. 验收标准

1. 鼠标点在模型身体可见区域上时，能够稳定触发拖拽
2. 鼠标点在模型外部透明背景上时，不会再凭空拖动模型
3. 模型部分移出屏幕后，只要仍有可见身体留在屏幕内，用户就能抓住可见部分把模型拖回
4. 切换至少两套留白差异明显的 Live2D 皮套后，不需要重新配置命中矩形
5. 命中改造后，聊天框交互不受破坏
6. 命中改造后，滚轮缩放、点击动作、多屏拖拽等已有交互不出现明显回归
7. 开启调试模式后，可追踪命中点坐标、Alpha 值与最终判定结果
8. 关闭 `ui.adaptive_hit_test_enabled` 后，行为可回退为旧逻辑
